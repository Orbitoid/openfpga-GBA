# SDRAM Timing & Internal Clock Closure Plan

## Background

A trusted FPGA professional identified that the SDRAM clock was being driven
through an ALTDDIO_OUT DDR primitive, which hides the clock path from the
Quartus timing analyzer. The original MiSTer design does this too — it feeds
the 0deg system clock into the DDR primitive, producing a 180deg SDRAM clock
with no timing analysis coverage.

During the Pocket port, a 270deg PLL output was added with the intent of
feeding it into the DDR primitive (270 + 180 = 90deg net), but the wiring
was never completed — `clk_sys_90` was generated by the PLL but never
connected to anything. The SDRAM clock was effectively 180deg via DDR on the
0deg system clock, identical to the MiSTer original.

We have now:
- Removed the ALTDDIO_OUT primitive
- Driven dram_clk directly from PLL outclk_1
- Determined through hardware testing that 270deg phase (7451 ps) works
  while the theoretically "ideal" 90deg (2484 ps) does not, due to the
  path delay mismatch between PLL-to-pin (~1 ns) and IOE-to-pin (~4 ns tCO)
- Added SDC constraints: create_generated_clock on dram_clk,
  set_output_delay for writes, set_input_delay + multicycle for reads
- Fixed SDC ordering bug: create_generated_clock now defined before
  set_clock_groups so the fitter sees sdram_clk during optimization
- Fixed CI to capture full STA/fitter reports and custom path-level reports
- Confirmed the timing analyzer now reports SDRAM margins for the first time

Current measured state (seed 7, 270deg PLL-direct, full SDC constraints):
- SDRAM write setup (slow 85C): **+0.041 ns** — barely passing
- SDRAM write hold (slow 85C):  **+0.973 ns** — comfortable
- SDRAM read path: multicycle-constrained, passing
- Internal system clock setup (slow 85C): **-1.197 ns**, TNS -22.455
- Internal system clock Fmax: 95.37 MHz (need 100.66 MHz)
- ALM utilization: 16,392 / 18,480 (89%)

A real user on a newer-generation Analogue Pocket reports Pokemon data
corruption mid-gameplay (name/level garbling, freezes, save corruption)
and German Pokemon Emerald crashing on boot. The core developer's older
Pocket does not reproduce these issues. This is consistent with marginal
timing that varies across hardware revisions.

## Completed Work

### SDRAM clock architecture (done)
- Removed ALTDDIO_OUT DDR primitive from `sdram_pocket.sv`
- Added `clk_sdram` input port, wired to PLL outclk_1 in `core_top.sv`
- Direct assignment: `assign dram_clk = clk_sdram`
- Hardware tested at 270deg — ROMs load, gameplay works on dev Pocket

### SDC timing constraints (done)
- `create_generated_clock -name sdram_clk` on `dram_clk` port
- `set_output_delay` for write path (tDS=1.5ns, tDH=0.8ns)
- `set_input_delay` for read path (tAC=6.0ns, tOH=2.5ns)
- `set_multicycle_path` setup 2 / hold 1 for sdram_clk→sys read capture

### SDC ordering fix (done)
- **Bug found:** `set_clock_groups` referenced `sdram_clk` before
  `create_generated_clock` defined it. Fitter warning (332174) confirmed
  it was ignoring the SDRAM clock group during timing-driven placement.
- **Fix:** Moved `create_generated_clock` above `set_clock_groups` in
  `core_constraints.sdc`. Next build will show the impact.

### CI report infrastructure (done)
- Both `build.yml` and `build-branch.yml` now upload:
  - `ap_core.sta.rpt` (full STA report)
  - `ap_core.fit.rpt` (full fitter report)
  - `ap_core.fit.summary`, `ap_core.sta.summary`, `ap_core.flow.rpt`
  - `build_output/reports/` (custom detailed path reports)
- Reports upload with `if: always()` so they're available even on failure
- Diagnostic step logs whether custom STA reports were generated
- Custom STA script (`sta_custom_report.tcl`) rewritten to use standalone
  `quartus_sta -t` mode instead of broken `--report_script` flag
- `generate.tcl` updated with file existence verification after STA run

## Step 1: Identify the failing internal paths

**Status:** Blocked — awaiting first build with custom STA fix

**Goal:** Determine exactly which register-to-register paths are responsible
for the -1.197 ns system clock setup violation.

**Why this is first:** The STA summary only reports worst-case slack per clock
domain. We need the full timing report to see whether the failing paths are
clustered in one module (e.g., SDRAM controller, GBA CPU, bus arbiter) or
scattered across the design. This determines whether the fix is targeted
pipelining or a broader utilization/placement problem.

**How:**
1. Run a CI build with the updated `generate.tcl` and `sta_custom_report.tcl`
   — this should produce `build_output/reports/ap_core.sta.paths_setup.rpt`
   with 80 worst setup paths and full path detail
2. The SDC ordering fix is included in this build, so the fitter will have
   correct clock groups. Compare system clock slack to the previous -1.197 ns
   to measure the impact of proper SDRAM clock grouping
3. From the setup paths report, document the top 10-20 failing paths:
   source register, destination register, data path delay, clock skew,
   and which RTL module they belong to
4. Look for patterns:
   - Are they all in one module? → targeted fix
   - Are they on paths to/from block RAM? → RAM output register issue
   - Are they crossing between VHDL (gba_top) and SV (Pocket wrappers)? →
     interface pipelining opportunity
   - Are they in the SDRAM controller? → directly explains user bug report

**Acceptance criteria:** A list of the critical failing paths with enough
detail to decide where to intervene.

## Step 2: Reduce ALM utilization

**Goal:** Get utilization below ~85% to give the fitter room to optimize.

**Why:** At 89% (16,392 / 18,480 ALMs), Quartus cannot move logic around
without creating new timing violations elsewhere. The seed sweep variation
of 0.5 ns between best and worst seeds confirms the fitter is struggling
with placement. Every subsequent optimization step will be more effective
with headroom.

**How:**
1. Review the MiSTer GBA core for features not used on Pocket:
   - DDR3 SDRAM interface (ddr_sdram module) — MiSTer supports both SDR
     and DDR3; Pocket only has SDR
   - Third SDRAM channel (ch3 in MiSTer's sdram.sv, used for BRAM/save)
     if save is handled differently on Pocket
   - HPS I/O, OSD, and other MiSTer framework features that may have left
     vestigial logic
2. Check for duplicate or redundant synchronizers, especially on CDC paths
   between clock domains
3. Look at block RAM utilization (281/308 = 91%) — some distributed RAM
   implemented in ALMs might be convertible to block RAM, but RAM blocks
   are also tight
4. Check if the synthesis settings (already aggressive) are over-duplicating
   registers via PHYSICAL_SYNTHESIS_REGISTER_DUPLICATION — the fitter report
   shows retiming recovered 3328 ps and created 300 register duplicates;
   verify these are all helpful
5. Profile which modules consume the most ALMs using the fitter report's
   resource usage by entity section

**Acceptance criteria:** Utilization below 85%, or a documented assessment
of why further reduction isn't feasible without feature cuts.

## Step 3: Fix the critical internal timing paths

**Goal:** Close the -1.197 ns system clock setup violation, or reduce it
to the point where a seed sweep can close it.

**Why:** This is the most impactful issue for reliability. Internal setup
violations can cause the SDRAM controller to generate wrong addresses,
wrong commands, or corrupt data — which directly explains the user's bug
report. Even if the SDRAM I/O timing is perfect, wrong data sent to the
SDRAM means wrong data stored and read back.

**How** (depends on Step 1 findings, but likely approaches):
1. **Pipeline long combinational paths:** If the failing paths have large
   data arrival times (> 12 ns), insert a register stage to break the path
   into two shorter cycles. Common candidates:
   - Address decode logic between GBA CPU bus and SDRAM controller
   - Mux select paths in bus arbitration (multiple requestors to SDRAM)
   - Wide data path operations (32-bit EWRAM read/write assembly)
2. **Add output registers on block RAM:** If failing paths end at block RAM
   inputs, ensure the RAM's registered input mode is enabled. If paths
   start at block RAM outputs, add an explicit output register
3. **Retime across module boundaries:** The Pocket wrapper (core_top.sv)
   bridges between gba_top.vhd and the SDRAM controller. If signals cross
   this boundary with long combinational paths, adding a pipeline register
   at the boundary can help without changing the core GBA logic
4. **Review the SDRAM controller state machine:** The sdram_pocket.sv
   controller uses combinational logic for next-state and output decoding.
   If any of the failing paths are in this module, register the outputs
   and adjust the state machine timing by one cycle

**Acceptance criteria:** System clock worst-case setup slack > -0.3 ns
(closable by seed sweep), or ideally positive.

## Step 4: SDRAM clock phase sweep

**Goal:** Find the optimal PLL phase for maximum SDRAM write setup margin.

**Why:** The current 270deg phase gives +0.041 ns write setup margin —
effectively zero. Different Pocket hardware revisions will shift this by
hundreds of picoseconds in either direction. More margin means fewer
hardware-dependent failures.

**How:**
1. Build with phase values: 240deg (6621 ps), 255deg (7036 ps),
   270deg (7451 ps, current), 285deg (7866 ps), 300deg (8278 ps)
2. For each build, record:
   - `sdram_clk` setup slack (slow 85C) — write path
   - `sdram_clk` hold slack (slow 85C) — write path
   - System clock setup slack — verify no regression
3. Plot or tabulate the results to find the phase that maximizes the
   minimum of (setup margin, hold margin)
4. If possible, test each build on hardware to verify ROM loading and
   gameplay stability

**Note:** Phase only affects the PLL configuration bits, not logic placement.
But with SDC constraints now in place, different phases will cause the fitter
to make different tradeoffs, so the sweep should use the same seed.

**Acceptance criteria:** SDRAM write setup margin > 0.3 ns in slow 85C model.

---

## Step 5: Seed sweep on final configuration

**Goal:** Find the best fitter seed for the optimized design.

**Why:** At high utilization, seed selection has significant impact (we've
seen 0.5 ns variation). After completing the above steps, a seed sweep on
the final RTL and constraints will find the best achievable timing.

**How:**
1. Run seeds 1-10 (or more if compute time allows)
2. Record system clock slack and SDRAM clock slack for each
3. Select the seed with the best worst-case across both domains
4. Hardware test the best seed build

**Acceptance criteria:** All timing domains meeting their targets from
Steps 3 and 4, or the best achievable with documented remaining violations.

---

## Step 6: Hardware validation on multiple Pocket revisions

**Goal:** Confirm the timing improvements resolve the user-reported
data corruption.

**Why:** Timing analysis is a model. Real hardware has additional factors
(temperature, voltage droop, crosstalk, PCB variation) that models
approximate but don't fully capture. Hardware testing on the specific
failing hardware revision is the final validation.

**How:**
1. Build the optimized core with best seed
2. Test on the developer's older Pocket — regression check
3. Send to the affected user (newer Pocket) for testing:
   - Pokemon game that previously showed name/level corruption
   - German Pokemon Emerald that previously crashed on boot
   - Extended play session (30+ minutes) to stress thermal margins
4. If issues persist, revisit Phase sweep (Step 4) with wider range,
   or investigate whether the read path multicycle assumption needs
   refinement

**Acceptance criteria:** No data corruption or crashes across both
Pocket hardware revisions over extended testing.
